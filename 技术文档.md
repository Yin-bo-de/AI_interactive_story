技术设计文档：幻境回响（PC端网页）
1. 技术架构
采用 现代前端框架 + 后端服务 + 大模型API 的架构。

前端：React 或 Vue.js 单页应用，部署于Vercel/Netlify或自建服务器。UI设计适配PC端宽屏，强调沉浸式阅读和输入体验。

后端：Node.js (Express/NestJS) 或 Python (FastAPI)，负责会话管理、上下文存储、与LLM API交互、计时与轮次控制。部署于云服务器。

数据库：可选，用于存储用户历史记录、数据统计。初期可用内存缓存或简单文件存储。

AI模型：接入当前顶级的推理/对话大模型API（如OpenAI GPT-4、Anthropic Claude、智谱AI等），利用其强大的上下文理解和生成能力。

2. 核心机制设计
2.1 故事生成与控制机制
为了平衡“完全现场生成”的自由度和“半小时内收尾”的可控性，采用 “秘密结局种子” 机制：

开局时：AI根据用户选择的类型，生成故事背景和开头，同时在系统内部（不展示给用户） 随机生成3个可能的“结局种子”。

例如：结局种子A（凶手是管家）、结局种子B（凶手是死者私生女）、结局种子C（密室是意外导致）。

推进时：AI根据用户的每一次输入，选择最合适的一个种子进行靠拢，逐步丰满线索和伏笔。

结局时：AI根据用户互动过程中实际选择的路径（即最常靠拢的那个种子），生成最终的、符合逻辑的故事结局。

2.2 上下文管理
全量+滑动窗口：后端维护该会话的完整对话历史。但由于模型上下文限制，发送给API时采用“关键摘要 + 最近N轮对话”的策略。

关键信息存储：将故事背景、出场人物、已发现的线索、预设的结局种子等关键信息，作为固定的“系统提示”的一部分，每次请求时都发送给模型。

结构化数据：后端可以维护一个JSON格式的“剧情状态”，随着互动更新，并作为上下文的一部分提供给模型。

2.3 计时与轮次控制
计时器：用户开始故事（收到AI第一段回复）时，后端启动30分钟倒计时。计时器独立于前端，防止用户刷新页面重置。

轮次计数器：每完成一次“用户输入->AI回复”的循环，计数器+1。达到100轮时，后端主动触发结束流程。

3. AI模型Prompt设计
以下是为AI角色设计的核心Prompt框架，分为“系统设定”和“每轮回复”两部分。

3.1 开局Prompt（初始化故事）
系统设定：

text
# 角色
你是一个顶尖的互动叙事AI，擅长创作悬疑推理故事。你现在将扮演故事中的【探长·林默】。

# 任务
1.  根据用户选择的“悬疑推理”类型，创作一个现代都市背景的密室杀人案开头。
2.  在开头明确你的角色身份（探长），并简要描述现场（死者、环境）。
3.  在系统内部（不要告诉用户）秘密生成三个可能的结局种子，用<seed>标签包裹，放在你的思考过程中（不输出给用户）。
    -   <seed>凶手是死者的商业伙伴，动机是财务纠纷</seed>
    -   <seed>凶手是死者的配偶，动机是情感背叛</seed>
    -   <seed>这是一场意外，被伪装成谋杀</seed>
4.  你的语气要冷静、略带疲惫，符合一个资深探长的形象。
5.  开头的最后，你需要向用户（你将称呼他为“搭档”或“顾问”）提出一个问题或请求，引导他进行下一步输入。例如：“搭档，你怎么看？我们先从哪开始查？”

# 约束
-   不要替用户做决定。
-   开头控制在300字以内。
3.2 互动推进Prompt（每轮用户输入后）
系统设定：

text
# 核心指令
你是一名正在办案的【探长·林默】。你的搭档（用户）刚刚说了一句话或做了一个行动。请以林默的身份进行回应，并推动剧情发展。

# 上下文信息
-   当前剧情摘要：[此处插入后端更新的剧情摘要，包含已发现的关键线索、已询问过的人物]
-   秘密结局种子：
    -   种子1：凶手是死者的商业伙伴，动机是财务纠纷
    -   种子2：凶手是死者的配偶，动机是情感背叛
    -   种子3：这是一场意外，被伪装成谋杀
-   本轮用户输入：[用户输入的文本]
-   对话历史：[最近5-10轮的对话记录]

# 回复要求
1.  【角色扮演】必须始终以林默的口吻回复，结合现场情况和你的经验。
2.  【剧情推进】根据用户输入，判断其最可能导向哪个种子，然后向该种子靠拢，自然地给出新的线索或信息。线索可以是指纹、证词、物品、时间线矛盾等。
3.  【互动引导】在回复的末尾，提供1-3个简短的下步行动建议，供用户选择。建议放在单独一行，用“>”或数字列表标识，增强可读性。
    -   *例如：*
        > 1. 去问问死者的秘书
        > 2. 去验尸房看看结果
        > 3. 你有其他想法
4.  【温和纠正】如果用户输入完全脱离推理范畴（如魔法、外星人），用符合角色设定的方式巧妙拉回。
    -   *例如（无奈地笑）：“搭档，科幻片改天再看，咱们手头这个案子还压在桌上呢。说回这个脚印……”*
5.  【创作配合】如果用户输入了合理的角色设定或情节走向（如“我怀疑秘书和死者有私情”），如果这个设定不与所有种子冲突，则采纳并融入剧情，甚至可以将其作为新的线索。

# 输出格式
-   直接输出林默要说的话，不要包含任何元信息。
3.3 游戏结束Prompt
系统设定：

text
# 任务
游戏时间已到（或互动轮次已达上限）。作为探长林默，你需要根据你们共同经历的调查过程，以及最终走向的那个秘密结局种子，为搭档生成一个完整的、合理的案件结局。

# 信息
-   秘密结局种子：最终选定的种子是[种子X]
-   整个调查过程摘要：[后端提供的剧情摘要，包含关键线索和用户的选择]

# 回复要求
1.  以林默的口吻，向搭档总结案件真相。可以从证据链、动机、手法等方面展开。
2.  结局必须逻辑自洽，并呼应之前互动中出现的关键线索。
3.  最后，对搭档（用户）的帮助表示感谢，并可以简单回顾一两个用户做出的关键贡献。
4.  篇幅适中，300-500字。
5.  在结局后，单独一行展示【分享结局】、【重新开始】的指引。

# 输出格式
-   直接输出林默要说的话。
4. UI/UX设计要点（PC端）
布局：采用三栏或两栏布局。中央主区域为对话流，左侧可展示故事状态（剩余时间/轮次、已发现线索、人物关系），右侧或底部为输入区域和选项卡片。

输入体验：大输入框，支持回车发送，输入字数实时计数（200字上限）。支持Shift+Enter换行。

沉浸感：可选用深色模式作为默认主题，字体选择适合阅读的衬线或非衬线体。AI头像和用户头像区分明显。

进度提示：顶部进度条或环形倒计时，视觉上提醒用户剩余时间。

5. 后续迭代方向
增加故事类型：在验证推理类型成功后，增加爱情、武侠、江湖等其他类型，前端增加类型切换的动画效果。

用户系统：增加登录注册功能，保存用户的历史故事和成就。

多角色切换：允许用户在故事中切换扮演不同角色，前端通过角色卡片切换。

社区与分享：用户可以分享自己生成的结局海报（前端生成图片）或链接到社交平台。
